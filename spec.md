# 天池移动推荐赛需求大纲（spec）

## 1. 项目目标
- 目标：基于 `2014-11-18` 到 `2014-12-18` 的用户行为，预测用户在 `2014-12-19` 对商品子集 `P` 的购买行为（`behavior_type=4`）。
- 交付：输出两列 `user_id` 与 `item_id`，以 `tab` 分隔，用于线上评测。
- 评估：以 `F1` 为唯一指标（兼顾 precision 与 recall）。

## 2. 业务与技术边界
- 推荐对象：只允许预测商品子集文件中出现的 `item_id`。
- 可用行为：点击(1)、收藏(2)、加购(3)、购买(4)。
- 时间粒度：小时级时间戳（`yyyy-MM-dd HH`）。
- 数据特点：
- 用户行为文件是全商品全集行为，不仅限于子集 `P`。
- 商品子集文件中 `item_id` 可能重复（同商品多地理位置）。
- 用户与商品 geohash 允许为空。
- 同一小时完全重复行为是合法数据，不直接判脏。

## 3. 本地数据确认（D:\tianchi-competition）
- `tianchi_fresh_comp_train_user_online_partA.txt`
- `tianchi_fresh_comp_train_user_online_partB.txt`
- `tianchi_fresh_comp_train_item_online.txt`
- 文件格式：无表头，TAB 分隔。
- 行字段推断：
- 用户行为：`user_id item_id behavior_type user_geohash item_category time`
- 商品子集：`item_id item_geohash item_category`

## 4. 任务拆解总览
- 阶段A：数据审计与样本构建
- 阶段B：特征工程与候选集生成
- 阶段C：模型训练与离线验证
- 阶段D：全量推断与结果导出
- 阶段E：复盘与可复现实验沉淀

## 5. 离线验证方案（先于实现确认）
- 验证原则：严格时间切分，避免信息泄漏。
- 推荐验证窗：
- 训练窗：`2014-11-18 ~ 2014-12-16`
- 验证标签日：`2014-12-17` 或 `2014-12-18`（可做滚动双验证）
- 线上模拟：训练到 `2014-12-18`，预测 `2014-12-19`
- 指标：
- 主指标：F1
- 辅助指标：Precision、Recall、预测集合规模、覆盖用户数

## 6. 建模策略（规划版）
- Baseline 1（可快速打分）：
- 规则召回：近N天有点击/收藏/加购的 `(user,item)`，按行为加权排序截断。
- 目标：先跑通端到端流程，建立可提交版本。
- Baseline 2（推荐主线）：
- 两阶段方案：候选召回 + 排序学习（分类）。
- 排序模型优先级：`LightGBM/XGBoost > LogisticRegression`（按资源择优）。
- 标签定义：目标日购买记1，未购买记0。
- 候选样本：用户近7/14天交互过的商品 + 热门商品补充。

## 7. 关键特征清单（首版）
- 用户侧：
- 近1/3/7/14天各行为次数
- 最近一次交互距目标日小时差
- 行为活跃时段分布（昼夜/工作日）
- 商品侧：
- 近1/3/7天曝光、加购、收藏、购买计数
- 转化率特征（加购->购、收藏->购）
- 类目侧：
- 用户-类目偏好强度
- 类目热度与转化
- 交叉侧（user-item）：
- 最近一次交互类型与时间差
- 近N天交互频次、行为序列强度
- 是否已历史购买
- 地理侧（可选增强）：
- user/item geohash 前缀一致度（长度4/5/6）

## 8. 结果输出与提交规范
- 输出文件仅两列：`user_id\titem_id`
- 无表头
- 去重：同一 `(user_id,item_id)` 仅保留一条
- 排序：可不强制，但建议按 `user_id,item_id` 排序便于核验
- 文件命名建议：`submit_YYYYMMDD_HHMM.tsv`

## 10. 风险与应对
- 风险1：数据体量大导致单机读取慢
- 应对：分块读取、按天聚合、优先 Parquet 中间层
- 风险2：正负样本极不均衡
- 应对：负采样 + 类别权重 + 阈值调优
- 风险3：离线线上分布偏移
- 应对：滚动时间验证 + 多阈值试验
- 风险4：F1 对阈值敏感
- 应对：在验证集网格搜索最优阈值与提交量
